音视频网关对接指南

WebRTC（Web Real-Time Communication）是一种实时通信技术，它允许网络应用或站点在没有中间人的情况下建立点对点的连接。

                    sdpDescription = setStartBitrate(RtcParameter.getInstance().VIDEO_CODEC_H264, true, sdpDescription, RtcParameter.getInstance().videoStartBitrateKbps);
                }

        private PeerConnection createPeerConnection() {
            try {
                if (_factory == null) {
                    Exception e = new Exception("createPeerConnection _factory is nil");
                    Gateway.instance().onError(e);

                PeerConnection.RTCConfiguration rtcConfig = new PeerConnection.RTCConfiguration(ICEServers);
                rtcConfig.tcpCandidatePolicy = PeerConnection.TcpCandidatePolicy.DISABLED; //TCP候选策略控制开关 (禁用)
                rtcConfig.bundlePolicy = PeerConnection.BundlePolicy.MAXBUNDLE; //捆绑的策略  (最大约束)
                rtcConfig.rtcpMuxPolicy = PeerConnection.RtcpMuxPolicy.REQUIRE; //实时传输控制协议多路策略
                rtcConfig.keyType = PeerConnection.KeyType.ECDSA;  //加密类型
                rtcConfig.enableCpuOveruseDetection = false;
                return _factory.createPeerConnection(rtcConfig, this);
            } catch (Exception e) {
                Gateway.instance().onError(e);
            }
            return null;
        }

  • 网关回调介绍

    • CallbackOpenActivity
      • 单人模式下（isMulti=false） : 当执行 Gateway.instance().callInvite(”110”) 成功后，拨打方和接听方都会触发CallbackOpenActivity 回调。
      • 在 CallbackOpenActivity 回调中，客户端需要初始化 WEBRTC 一个 PeerConnection（代码参考前面WEBRTC SdpObserver, PeerConnection.Observer 中的关键代码） 并且获取音频和视频(isVideo=true)流, 并将音视频流都添加到 PeerConnection 中，等待服务端发起 WEBRTC OFFER。
      • 完成 WEBRTC 初始化和音视频流的添加之后, 执行Gateway.instance().updatePeer(Gateway.instance().number(),Gateway.PEER_STATUS_INCALL) 通知服务端，客户端已经完成WEBRTC 初始化和音视频流的添加，服务端会发起 WEBRTC OFFER 通过几次 Sdp 交互之后服务端和客户端的 PeerConnection 状态都会变成 connected 服务端会收并保存到对应客户端音视频流。
      • 最后因为在 CallbackOpenActivity 回调的时候网关会开启一个异步锁(如果客户端没有响应，1s 后会自动超时释放)，所以客户端必需要在最后执行Gateway.instance().doneViewOpen() 来释放网关的异步锁。
    • CallbackOpenMutilActivity
      • 多人模式下（isMulti=true） : 客户端触发 CallbackOpenMutilActivity 回调，流程和CallbackOpenActivity 回调一样。
      • 当第二个拨打邀请触发 CallbackOpenMutilActivity回调的时候不需要再初始化 WEBRTC 和添加音视频流，只需要在页面上新增一个客户端信息（等待接听或者挂断）。
    • CallbackAnswer
      • 当执行 Gateway.instance().callAnswer(”110”) 成功后,拨打方和接听方都会触发CallbackAnswer 回调,Gateway.instance().updatePeer(Gateway.instance().number(),Gateway.PEER_STATUS_ANSWER) 通知服务端，客户端已经收到接听邀请。
      • 服务端会再次发起 WEBRTC OFFER 并将前面保存的音视频流发送给对方，对应客户端的 PeerConnection 的 onAddTrack 监听器会触发并收到对应的音视频流，此时就可以播放声音和画面了 。
    • CallbackHold
      • 当执行 Gateway.instance().callHold(”110”) 成功后, 拨打方和接听方都会触发CallbackHold 回调。
      • 服务端会发起 WEBRTC OFFER 移除双方音视频流的推送，对应客户端的 PeerConnection 的 onRemoveTrack PeerConnection 的 onRemoveTrack监听器会触发通知客户端移除对应通话者的画面,Gateway.instance().updatePeer(Gateway.instance().number(),Gateway.PEER_STATUS_HOLD) 通知服务端，客户端已经进入保持状态。
    • CallbackHangUp
      • 服务端可以执行 Gateway.instance().callHangupAll("挂断") 挂断所有通话者,也可以执行 Gateway.instance().callHangup(“1110”，"挂断") 挂断特定的通话者。
      • 单人模式下（isMulti=false）
        • 拨打方和接听方都会触发CallbackHangUp 回调，需要在回调中释放摄像头和话筒，并且断开 WEBRTC PeerConnection 最后关闭页面。
      • 多人模式下（isMulti=true)
        • 当通话人数大于 1 的时候（不包括服务端），单个通话者挂断之后只有对应的客户端会触发CallbackHangUp 回调，服务端 PeerConnection 的 onRemoveTrack监听器会触发通知客户端移除对应通话者的画面，其他通话中的用户不会受到影响。
        • 当通话人数小于等于 1 的时候，拨打方和接听方都会触发CallbackHangUp 回调，需要在回调中释放摄像头和话筒，并且断开 WEBRTC PeerConnection 最后关闭页面。
      • 最后因为在 CallbackHangUp 回调的时候网关会开启一个异步锁(如果客户端没有响应，1s 后会自动超时释放)，所以客户端必需要在最后执行Gateway.instance().doneCloseOpen() 来释放网关的异步锁。
    • CallbackBroadcast
      • 当执行 Gateway.instance().broadcast(”110”,"broadcast json data") 成功后, 110 号码客户端会触发 CallbackBroadcast 回调。
      • 第一个参数为号码，第二个参数通常是json字符串。
    • CallbackReceiverSdp
      • 配合 Gateway.instance().sendSdp() 使用来进行信令信息交换，信令交换也有异步锁，需要执行Gateway.instance().doneSdp() 来释放异步锁。
    • CallbackSuccess
      • 网关启动成功后触发，通知客户端网关已经启动。
    • CallbackError
      • 网关发生错误时候触发，用于发送网关错误信息给客户端。
    • CallbackNotice
      • 网关发生非致命错误时候触发，用于发送网关警告信息给客户端。
